# Agents

Agents let p2pstream forward public traffic into networks that do not accept inbound connections.

## What It Is

An agent is a registered remote worker with a generated public ID and token. It connects outbound to the management server over HTTPS/WSS, receives selected requests, forwards them from its own network, and streams responses back.

## When It Matters

Use agents for home lab services, private network apps, remote hosts behind NAT, or backend pools that need more than one private network path.

## Runtime Behavior

Agent IDs and tokens are generated by management. Tokens are shown once when created or rotated and are stored server-side as hashes.

Agents connect outbound to management:

- HTTPS for stats and API calls,
- WSS for request forwarding.

The agent and server send periodic WebSocket heartbeat pings so stale or idle intermediary connections are detected before traffic depends on them. The default ping interval is 20 seconds with a 10 second ping timeout. If another reverse proxy sits in front of the management listener, it must allow WebSocket upgrades and its idle timeout should be greater than the heartbeat interval.

By default, agents reject insecure HTTP management URLs and verify HTTPS certificates. For auto-generated management TLS, pass one of:

| Variable | Use |
| --- | --- |
| `MANAGEMENT_CA_FILE` | Path to a PEM CA bundle on the agent host. |
| `MANAGEMENT_CA_PEM_BASE64` | Base64-encoded PEM CA bundle, useful for generated snippets. |

The Agent Setup dialog can generate a one-line Linux systemd installer, a Docker Compose service, or a direct CLI command. The Linux installer downloads the release binary, verifies the checksum, writes `/etc/p2pstream/agent.env`, and enables `p2pstream-agent.service`.

If management requires agent client certificates, configure:

```text
AGENT_TLS_CERT_FILE=/etc/p2pstream/agent.crt.pem
AGENT_TLS_KEY_FILE=/etc/p2pstream/agent.key.pem
```

## Common Mistakes

- Reusing an old token after rotating it in management.
- Setting `MANAGEMENT_URL` to a public listener instead of management.
- Putting management behind a reverse proxy that blocks WebSocket upgrades or closes idle WebSockets too aggressively.
- Forgetting CA material when management uses the auto-generated local CA.
- Deleting the management agent record and expecting files to be removed from the remote host.
